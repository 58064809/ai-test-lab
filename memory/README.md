# 记忆系统说明

## 一、定位

本目录是 `ai-test-lab` 的项目记忆层。

这里不自研记忆系统，也不一开始就上复杂向量库。第一阶段采用工业界常见的轻量方式：

```text
结构化 Markdown 文件 + Git 版本管理 + AI 工具按需读取
```

这样可以同时被 ChatGPT、Codex、OpenHands、Claude Code、PyCharm、命令行工具读取和复用。

## 二、核心原则

1. 不造轮子。
2. 不把记忆写死在某一个 AI 工具里。
3. 记忆必须可读、可改、可版本管理、可迁移。
4. 先文件化，再根据需要升级到 RAG 或向量库。
5. 只沉淀长期稳定的信息，不把临时聊天内容全部塞进记忆。
6. 记忆内容默认中文，除非技术名词、命令、代码、配置项必须使用英文。

## 三、记忆分层

```text
memory/
├── README.md                 # 记忆系统说明
├── project-memory.md         # 项目目标、边界、当前方向
├── user-preferences.md       # 用户偏好、输出规则、工作方式
├── testing-rules.md          # 通用测试规则和质量标准
├── ecommerce-domain.md       # 电商业务领域知识
└── memory-management.md      # 记忆维护规则
```

## 四、读取优先级

当 AI 处理任务时，建议按以下顺序读取：

```text
1. memory/project-memory.md
2. memory/user-preferences.md
3. 与任务相关的 skills/*.md
4. 与任务相关的 templates/*.md
5. 与任务相关的 workflows/*.md
6. 与任务相关的 tools/*.md
```

## 五、什么应该进入记忆

适合进入记忆：

- 项目长期目标
- 用户稳定偏好
- 测试用例输出格式
- 缺陷报告格式
- 不造轮子原则
- 常用测试方法
- 电商领域稳定业务知识
- 自动化框架设计偏好
- 常用工具选型

不适合进入记忆：

- 一次性聊天内容
- 未确认的猜测
- 临时账号密码
- 敏感密钥
- 无复用价值的细节
- 过期业务规则
- 未验证的技术方案

## 六、后续升级路径

第一阶段：Markdown 文件记忆。

第二阶段：让 OpenHands、Codex、Claude Code、PyCharm 等工具直接读取这些文件。

第三阶段：当文件数量变多后，再考虑接入现成 RAG 工具或向量库。

第四阶段：如果需要自动检索，再考虑 LangChain / LlamaIndex 等成熟框架。

## 七、结论

当前阶段的记忆系统不是数据库，也不是平台，而是一个可版本管理的个人测试知识资产库。

它的目标是让 AI 在执行任务前知道：

```text
我是谁
我要做什么
我不做什么
我怎么输出
我有哪些测试规则
我有哪些业务上下文
我应该优先使用哪些成熟工具
```
